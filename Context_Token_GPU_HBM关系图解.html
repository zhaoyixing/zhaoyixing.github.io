1 Context Size 与 Token：从文字到数字

当你向大模型发送一段话时，模型不是"阅读文字"，而是把文字切分成Token——可以理解为模型世界里的"最小单词"。

🇨🇳中文：1个汉字 ≈ 1.5-2个Token
"今天天气真好" (6字) → 约 8-10 Token
中文因BPE编码效率较低，Token密度高于英文
🇬🇧英文：1个单词 ≈ 1-1.3个Token
"The weather is nice today" (5词) → 约 5-6 Token
常见词通常1个Token，长词可能拆成2-3个
Context Size = 输入Token + 输出Token
Context Window (200K) = Prompt (你发的) + Completion (AI回的)
例如：Claude Opus 4.6 的 Context Window = 200K tokens ≈ 约10-15万个汉字 ≈ 一本中篇小说

2 关键中间层：KV-Cache（显存杀手）

这是最关键的环节。模型在生成每一个新Token时，需要"回顾"之前所有Token的信息。这些信息存储在KV-Cache中——可以想象成模型的"短期记忆"。

KV-Cache 显存计算公式
KV-Cache (bytes) = 2 × n_layers × n_heads × d_head × seq_len × dtype_size
2 = K和V两个矩阵 | n_layers = 模型层数 | seq_len = Context长度 | dtype = FP16(2B) 或 FP8(1B)

4 GPU与HBM：推理的"物理天花板"

所有Token的计算、所有KV-Cache的存储，最终都发生在GPU的HBM（高带宽内存）上。HBM决定了两件事：

📦容量 = 能装多少
HBM容量决定了GPU能同时容纳多大的模型权重 + 多少用户的KV-Cache。
80GB H100 → 勉强装下70B模型+少量KV-Cache
288GB GB300 → 可装200B模型+大量并发KV-Cache
🚀带宽 = 能跑多快
推理本质是"读取权重 → 计算 → 输出Token"。Token生成速度 ≈ HBM带宽 ÷ 模型大小。
3.35 TB/s H100 → 70B模型约 24 T/s/user
12 TB/s GB300 → 70B模型约 86 T/s/user

GPU代际演进：HBM容量 × 带宽 × 推理效率

每一代GPU的HBM显存增长如何直接提升推理能力
为什么推理是"Memory-Bound"而非"Compute-Bound"？
推理（生成Token）不像训练那样需要大量矩阵乘法。推理的瓶颈在于：每生成1个Token，都需要从HBM中完整读取一遍模型权重。

算力利用率 H100推理时GPU算力利用率仅 1-5%，99%的时间在等HBM "喂数据"。
所以 HBM带宽是推理速度的决定性瓶颈 → 这就是为什么每代GPU都在拼HBM升级。
5 链路放大效应：1个Token背后的连锁反应
⚡ 放大效应总结：Context翻倍 → Token翻倍 → KV-Cache翻倍 → 并发减半 → GPU需求翻倍以上。这就是为什么Agent时代（平均Context从2K→8K，增4倍）对GPU/HBM的需求不是4倍，而是6-8倍——因为还有带宽瓶颈和batch效率下降的复合效应。
