案例3：负载均衡与有状态/无状态服务

Load Balancing & Stateful/Stateless Services

攻克：LVS/Nginx/HAProxy · 负载算法 · 状态服务判断 · 水平扩展

案例原题回顾

📋 案例背景：某分布式 Web 系统的负载均衡与服务构件状态设计2026-04-20 得分：17/25

问题 1（8 分）：填空题。① 列举 2 种软件负载均衡的常见实现；② 相同来源请求分到同一服务器的静态算法；③ 根据服务器性能差异分配的静态算法。

问题 2（6 分）：说明"加权最小连接数"算法的适用场景及其与普通"最小连接数"的区别。

问题 3（11 分）：系统有 5 种服务构件——(a) 身份认证、(b) 资源发布、(c) 资源检索、(d) 在线编辑、(e) 统计分析，要求：① 分类为有状态/无状态；② 解释为什么统计分析要设计为无状态。

失分主因：① 软件负载均衡误写"DNS 负载均衡"（DNS 属于全局负载均衡）；② "资源发布"误判为有状态（应为无状态）。

考点精讲

🔴 超高频必背
负载均衡的两大分类

按部署位置分类：

  • 软件负载均衡：部署在服务器上的软件，如 LVS、HAProxy、Nginx
  • 硬件负载均衡：专用设备，如 F5、A10
  • 全局负载均衡（GSLB）：跨地域，如 DNS 负载均衡、智能 DNS

⚠️ 踩雷：案例题问"软件负载均衡的常见实现"时，DNS 是错的！DNS 属于全局负载均衡。

🔴 超高频必背
软件负载均衡三件套
  • LVS（Linux Virtual Server）
    四层（传输层 TCP/UDP）
    性能最高，不解析 HTTP
  • Nginx
    七层（应用层 HTTP）
    最灵活，支持动静分离、反向代理
  • HAProxy
    兼顾四层 + 七层
    高并发 TCP 代理首选

记忆：LVS 四层最快 / Nginx 七层最灵 / HAProxy 兼得

🔴 超高频必背
负载均衡算法（静态 + 动态）

静态算法（不看实时状态）：

  • 轮询（RR）：依次分配
  • 加权轮询（WRR）：按性能权重分配
  • 随机（Random）：随机选择
  • 源地址哈希（IP Hash）：相同客户端→同一服务器（会话保持）

动态算法（看实时状态）：

  • 最小连接数（LC）：连接最少的服务器优先
  • 加权最小连接数（WLC）：连接数 + 权重
  • 最短响应时间：响应最快的优先
🔴 高频
加权最小连接数详解

适用场景：

后端服务器性能差异较大——比如部分服务器 CPU/内存配置更高，能处理更多请求。

与普通最小连接数的区别：

算法考虑因素问题
最小连接数只看当前连接数性能弱的服务器可能过载
加权最小连接数连接数 + 权重高配机器分担更多连接，实现"按能力分配"
🔴 超高频必背
有状态 vs 无状态服务（核心判断）

🎯 判断标准：服务端是否需要记住会话/进度/上下文？

类型特征典型例子
有状态服务端需维护会话/上下文身份认证（存 session）
在线编辑（存文档状态/锁）
无状态每次请求独立完成资源发布（一次存库即完成）
资源检索（请求独立）
统计分析

⚠️ 最常见踩雷：资源发布不是有状态！只要不需要服务端记住"我上次发到哪了"，就是无状态。

🔴 超高频必背
无状态服务的 3 大好处

案例题问"为什么设计为无状态"，必答三点：

  1. 可伸缩性强
    任意服务器都可处理任意请求，方便水平扩展和负载均衡
  2. 容错性好
    服务器重启/故障不会丢状态，可用性更高
  3. 避免状态同步复杂性
    无需跨节点同步 session 或上下文，降低开发/运维难度
🟡 中频
有状态服务的横向扩展方案

如果某服务必须有状态（如身份认证），如何解决水平扩展问题？

  • Session 集中存储：存入 Redis/Memcached（把状态外置）
  • Sticky Session（会话粘滞）：用 IP Hash 保持同客户端到同服务器
  • Token 无状态化：用 JWT 把状态编码进 token，转成"伪无状态"

💡 趋势：现代架构倾向把状态外置（Redis）或 Token 化，让服务尽可能无状态。

🟡 中频
七层 vs 四层负载均衡
四层（L4）七层（L7）
工作层传输层 TCP/UDP应用层 HTTP
转发依据IP + 端口URL / Header / Cookie
性能高较低
代表LVS、F5Nginx、HAProxy
场景TCP 长连接动静分离、URL 路由

记忆口诀

软件负载均衡三件套
LVS 四层快，Nginx 七层灵，HAProxy 兼顾两层（DNS 是全局，不算软件！）
负载均衡算法速记
静态：轮询、加权轮询、随机、IP 哈希；动态：最小连接、加权最小连接、最短响应
加权最小连接数适用
"服务器性能差异大"——高配机器分担更多连接
有状态 vs 无状态判断
要记会话/进度/上下文→有状态；一次请求完一件事→无状态
典型服务分类
有状态：认证、在线编辑；无状态：发布、检索、统计分析
无状态服务 3 好处
可伸缩 + 容错好 + 无同步复杂（"横向扩展无负担"）
有状态横向扩展方案
Session 外置（Redis）+ Sticky Session + JWT Token 无状态化

测试题（10题，自动判分）

Q1：以下哪个不是软件负载均衡的典型实现？
ALVS
BNginx
CHAProxy
DDNS 负载均衡
✓ 正确答案：D
DNS 负载均衡属于全局负载均衡（GSLB）的范畴，不是软件负载均衡的典型实现。软件负载均衡三件套是 LVS / HAProxy / Nginx。
Q2：需要让"来自同一 IP 的请求始终分到同一台服务器"，应选哪种算法？
A轮询（RR）
B加权轮询（WRR）
C源地址哈希（IP Hash）
D随机
✓ 正确答案：C
源地址哈希（IP Hash）对客户端 IP 计算哈希值来选择服务器，保证同一 IP 始终命中同一服务器，常用于简单会话保持。
Q3：加权最小连接数算法最适合以下哪种场景？
A所有服务器配置完全相同
B后端服务器性能差异较大
C服务器无状态且请求数低
D所有请求处理时间相同
✓ 正确答案：B
加权最小连接数在最小连接数基础上引入权重，让高配机器分担更多连接，适用于后端服务器性能差异大的场景。若配置相同，普通最小连接数就够用。
Q4：某系统有 5 种服务构件：身份认证、资源发布、资源检索、在线编辑、统计分析。下列分类正确的是？
A有状态：身份认证、资源发布；无状态：资源检索、在线编辑、统计分析
B有状态：身份认证、在线编辑；无状态：资源发布、资源检索、统计分析
C有状态：资源发布、在线编辑；无状态：身份认证、资源检索、统计分析
D全部是有状态
✓ 正确答案：B
身份认证维护 session → 有状态；在线编辑维护文档上下文/锁 → 有状态；资源发布、资源检索、统计分析都是一次请求完成，不依赖上下文 → 无状态。资源发布常被误判为有状态！
Q5：将服务设计为无状态的主要好处不包括？
A提升可伸缩性，便于水平扩展
B容错性好，服务重启不丢状态
C避免跨节点状态同步的复杂性
D保证强一致性
✓ 正确答案：D
无状态与"强一致性"无关。A/B/C 三点是教材标准的三大好处，必须背熟。
Q6：LVS 工作在 OSI 的哪一层？
A数据链路层（二层）
B网络层（三层）
C传输层（四层）
D应用层（七层）
✓ 正确答案：C
LVS 工作在传输层（四层），基于 TCP/UDP 转发，不解析 HTTP，性能最高。Nginx 主要工作在七层。
Q7：关于有状态服务的横向扩展方案，下列哪项不正确？
A将 Session 集中存入 Redis
B使用 Sticky Session（IP Hash）
C使用 JWT Token 把状态编码进 Token
D在每台服务器各自维护独立的内存 Session，不做同步
✓ 正确答案：D
独立内存 Session 不同步会导致用户请求被分到其他服务器时丢失登录状态，无法支持水平扩展。A/B/C 都是常用方案。
Q8：以下关于加权轮询（WRR）的描述正确的是？
A动态算法，看服务器实时连接数
B静态算法，根据服务器配置权重循环分配
C只能在 LVS 中实现
D会保证同一客户端请求到同一服务器
✓ 正确答案：B
加权轮询是典型静态算法，不看实时状态，按权重比例循环分配。同一客户端到同一服务器是 IP Hash 的特性。
Q9：Web 应用需要做动静分离（HTML 走静态 CDN、API 走后端），最适合的负载均衡工具是？
ALVS
BNginx
CF5
DDNS
✓ 正确答案：B
Nginx 工作在七层，能根据 URL/路径做动静分离、反向代理，是 Web 应用的标配。LVS 只做四层转发，无法看 URL。
Q10：下列哪个服务最适合被设计为"无状态"？
A在线文档协同编辑
B网游战斗实时同步
C订单数据统计分析
D视频会议房间管理
✓ 正确答案：C
订单统计分析每次请求独立（按时间范围查数据→聚合→返回），不依赖上下文，典型无状态。A/B/D 都需要维护实时会话/房间状态。
