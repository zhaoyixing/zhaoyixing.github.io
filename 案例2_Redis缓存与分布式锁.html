Redis Cache & Distributed Lock

考点精讲

🔴 超高频必背
Redis 5 大数据结构与场景
  • String（字符串）：简单 KV、计数器、分布式锁（SET NX）
  • Hash（哈希表）：存储对象属性（如用户信息 name/age/addr）
  • List（列表）：消息队列、最新消息 N 条
  • Set（集合）：用户标签/画像、去重、交并差集运算（分群）
  • ZSet（有序集合）：排行榜、带分数的排序

⚠️ 踩雷：用户标签场景别写 Hash！Hash 是"对象属性"不是"集合"，没有去重和集合运算语义。

🔴 超高频必背
用户画像为什么用 Set（不是 Hash）

画像标签场景的 3 个核心需求：

  1. 天然去重：同一标签不会被重复添加
  2. 快速判存在：SISMEMBER 时间复杂度 O(1)
  3. 集合运算分群：SINTER/SUNION/SDIFF 做"北京 AND 学生"

Hash 不适合的原因：

  • 没有天然去重
  • 不支持集合运算（交并差）
  • 语义不匹配（Hash 是"对象属性"）

✅ 扩展：如果标签带权重（喜欢度 0-100），用 ZSet；纯标签用 Set。

🔴 超高频必背
LRU：核心是"局部性原理"

LRU = Least Recently Used（最近最少使用）

考点三连问（案例填空）：

  • 算法基础是什么？→ 局部性原理（最近访问过的数据未来更可能再访问）
  • 淘汰哪些数据？→ 最近未（最少）被使用的数据
  • 对哪些 key 生效？→ 取决于策略（volatile-lru 只淘汰带过期时间的，allkeys-lru 淘汰所有）

⚠️ 踩雷：填空问"算法思想"时，答"LRU"本身是错的——要答"局部性原理"！

🔴 高频
Redis 8 种内存淘汰策略

按范围 × 策略分为 4×2 = 8 种：

  • noeviction：不淘汰，写入返回错误（默认）
  • volatile-lru / allkeys-lru：按 LRU 淘汰
  • volatile-lfu / allkeys-lfu：按 LFU（最不常用）淘汰
  • volatile-random / allkeys-random：随机淘汰
  • volatile-ttl：淘汰即将过期的 key

volatile-* vs allkeys-*：

前者只淘汰带过期时间的 key；后者淘汰所有 key（纯缓存首选 allkeys-lru/lfu）。

🔴 超高频必背
布隆过滤器（防缓存穿透）

解决的问题：缓存穿透（大量不存在的请求直接打到 DB）

原理：多个哈希函数 + bit 数组，快速判断"一定不存在"或"可能存在"。

3 大缺点（案例至少答 2 点！）：

  1. 存在误判率（假阳性）——可能把不存在的误判为存在
  2. 不支持删除元素——bit 位无法回退
  3. 只能判断有无，不能存储实际数据

⚠️ 踩雷：只答"只能判断有无"会被扣分！必须覆盖"误判率"和"无法删除"。

🔴 超高频必背
缓存三大问题（穿透/击穿/雪崩）
问题定义解决方案
穿透查不存在的 key布隆过滤器 / 缓存空值
击穿热点 key 过期瞬间被大量请求互斥锁 / 永不过期
雪崩大量 key 同时过期过期时间加随机 / 多级缓存

⚠️ 易混：穿透=不存在的 key；击穿=单点热 key；雪崩=大面积同时过期。

🔴 超高频必背
Redis 分布式锁核心要点

锁的 4 大特性：

  • 互斥性（同一时刻只有一个客户端持锁）⭐
  • 防死锁（设置过期时间）
  • 可重入性（同一客户端可重复获取）
  • 解锁安全性（只有持锁者能释放）

核心原子操作：

SET key value NX PX 30000

NX = 不存在才设置；PX = 过期时间毫秒。一条命令原子完成"加锁 + 过期"。

⚠️ 踩雷：锁的关键特性是"互斥性"，不是"一致性"！

🔴 高频
Redis 锁为什么不是强一致（AP 模型）

三大非强一致原因：

  1. 主从同步延迟：主节点加锁后，key 未同步到从节点就宕机 → 新主节点无锁 → 双客户端持锁
  2. Redis 是 AP 模型（最终一致），不像 ZK/etcd 是 CP 模型
  3. 时钟漂移：服务器/客户端时钟不同步导致过期判定不一致

强化方案（不能根治但可降低风险）：

  • RedLock 算法：多个独立 Redis 过半成功才算加锁
  • 看门狗续约：持锁期间自动续期防止误过期
  • Lua 脚本原子释放：判断持有者 = 自己才能释放锁

⚠️ 适用场景：秒杀/限流（AP可接受）；不适用：金融交易/资金扣减（需 ZK/etcd）。

🟡 高频
Redis 数据分片：一致性哈希

普通哈希分片的问题：

节点数变化时（N→N+1），几乎所有 key 的 hash(key)%N 都变化 → 大面积数据迁移。

一致性哈希的优势：

  • 把节点和 key 都映射到一个 0~2^32 的环上
  • 新增节点时，只影响环上"下一段"的 key
  • 数据迁移量最少（案例题采分关键词）

虚拟节点：解决数据倾斜（不均匀）问题

测试题（10题，自动判分）

Q1：某电商网站要为每个用户存储"标签集合"（如"男性""学生""北京用户"）并支持分群计算，最合适的 Redis 数据结构是？
AHash
BSet
CList
DString
✓ 正确答案：B
Set 天然去重、O(1) 判存在、支持 SINTER/SUNION/SDIFF 集合运算做分群，完美匹配用户画像场景。
Q2：LRU 淘汰策略的核心思想基于什么原理？
A随机原理
B先进先出原理
C局部性原理
D哈希均匀原理
✓ 正确答案：C
LRU 基于局部性原理——"最近访问过的数据，未来更可能再被访问"。这是案例填空的采分关键词。
Q3：布隆过滤器主要解决什么问题？
A缓存击穿（热点 key 过期）
B缓存雪崩（大量 key 同时过期）
C缓存穿透（查询不存在的 key）
D缓存一致性问题
✓ 正确答案：C
布隆过滤器快速判断"key 是否一定不存在"，拦截不存在的请求直接打到 DB，解决缓存穿透。
Q4：以下哪项不是布隆过滤器的缺点？
A存在误判率（假阳性）
B无法删除已添加的元素
C只能判断有无，不能存储实际数据
D查询时间复杂度高达 O(N)
✓ 正确答案：D
布隆过滤器查询时间复杂度为 O(K)（K 是哈希函数个数，固定），非常快。A/B/C 都是教材标准缺点。
Q5：Redis 分布式锁的关键特性是？
A一致性
B互斥性
C可用性
D持久性
✓ 正确答案：B
分布式锁最核心的特性就是互斥性（同一时刻只有一个客户端持锁）。一致性是 Redis 本身的弱项，不是锁的"特性"。
Q6：Redis 实现分布式锁的正确原子命令是？
ASET key value + EXPIRE key 30
BSET key value NX PX 30000
CSETEX key 30 value
DGETSET key value
✓ 正确答案：B
SET key value NX PX 30000 在一条命令中原子完成"不存在才设置 + 设置过期时间"。选项 A 分两条命令，非原子；C 无 NX；D 无法加过期。
Q7：Redis 分布式锁不能保证强一致性的原因不包括？
A主从同步延迟
BRedis 是 AP 模型
C时钟漂移
DSETNX 不是原子操作
✓ 正确答案：D
SETNX/SET NX 是原子操作（单节点内）。非强一致的原因是 A/B/C 三者，不包括 D。
Q8：扩容集群节点时，以下哪种分片方案能使数据迁移最少？
A按 key 模 N 的普通哈希分片
B按 key 范围分片（a-m / n-z）
C一致性哈希
D随机分配
✓ 正确答案：C
一致性哈希通过环形结构，扩容时只影响环上下一段数据，迁移量最少。普通哈希扩容会导致大面积重分布。
Q9：某系统需要实现"销售排行榜 Top 10"，最佳 Redis 数据结构是？
AList
BSet
CHash
DZSet（有序集合）
✓ 正确答案：D
ZSet 带分数（销售额），支持 ZRANGE/ZREVRANGE 快速取 Top N，是排行榜场景的标准选择。
Q10：关于 volatile-lru 和 allkeys-lru 的区别，正确的是？
A两者淘汰算法完全不同
Bvolatile-lru 只淘汰带过期时间的 key；allkeys-lru 淘汰所有 key
Callkeys-lru 只淘汰永久 key；volatile-lru 淘汰所有 key
D两者的适用场景完全相同
✓ 正确答案：B
volatile-* 只对带过期时间的 key 生效；allkeys-* 对所有 key 生效。算法（LRU）本身相同，仅作用范围不同。纯缓存首选 allkeys-lru。
