1可修改性的四个方面
定义：能够快速地以较高性能价格比对系统进行变更的能力。通常以某些具体的变更为基准，通过考察这些变更的代价来衡量。

四个方面（必背 · 案例题填空高频）

方面含义举例
可维护性
Maintainability		修复缺陷、bug 修补线上 Bug 定位与修复
可扩展性
Extensibility		增加新功能、新需求电商加"直播"模块
结构重组
Restructuring		重组系统结构、模块化重构单体拆分为微服务
可移植性
Portability		移植到不同硬件/OS/平台从 Windows 移植到 Linux
记忆口诀：维 / 扩 / 重 / 移（修 bug、加功能、调结构、换平台）
⚠️ 易错
可修改性 ≠ 只修 bug。它是一个大伞：修 bug、加功能、重构、换平台 全部包含。
2零信任网络（Zero Trust Network）
核心思想：永不信任，始终验证
Never Trust, Always Verify

抛弃"内网可信、外网不可信"的传统边界假设。默认任何用户、设备、网络流量都不可信，每次访问都重新验证。

三大基本原则 高频

  1. 显式验证（Verify Explicitly）：每次访问基于身份+设备+位置+行为多维度做认证授权
  2. 最小权限访问（Least Privilege）：仅授予完成任务所需的最小、最短时效权限
  3. 假设已被攻破（Assume Breach）：默认网络已沦陷，做好横向防御与微隔离

关键技术组件

  • 强身份认证 IAM / MFA（多因素认证）
  • 动态 / 上下文感知授权（基于风险评分）
  • 微隔离 Micro-segmentation：限制东西向流量、阻止横向移动
  • 持续监控与行为分析（UEBA）
  • 端到端加密通信
  • 设备健康检查（Device Posture）
  • SDP 软件定义边界（Software Defined Perimeter）

零信任 vs 传统边界（含 SSL/TLS）

维度传统边界零信任
信任模型内网可信永不信任
防御重点抵御外部攻击防内部威胁 + 横向移动
控制粒度网络边界（IP/端口）每次访问、每个资源
授权方式静态、一次性动态、持续
适用场景传统数据中心云 / 混合 / 远程办公
📝 案例题答题模板：王工为何认为零信任更优？（4 分）
  1. 防内部威胁：SSL/TLS 假设内网可信，无法防范内鬼或已被渗透的账户；零信任对每次访问都验证
  2. 防横向移动：通过微隔离阻止攻击者在内网横向扩散
  3. 细粒度访问控制：基于身份+设备+上下文动态决策，而非仅靠 IP/端口
  4. 持续验证：会话期间持续监控，发现异常立即阻断（SSL/TLS 一次握手后不再校验）
  5. 降低整体风险面：默认拒绝 + 最小权限，缩小攻击面
3SSL/TLS 协议

基本定位

  • SSL（Secure Sockets Layer）= 安全套接字层，已淘汰
  • TLS（Transport Layer Security）= 传输层安全协议，SSL 的后继者
  • 工作位置：传输层与应用层之间（TCP 之上、HTTP 之下）
  • 常用版本：TLS 1.2 / 1.3（推荐 1.3）

三大功能 必背

口诀：认 / 密 / 整（身份认证、加密传输、完整性）
  1. 身份认证：基于数字证书 + CA 体系，验证服务端（可选客户端）身份
  2. 加密传输：非对称加密协商会话密钥 + 对称加密传输数据（混合加密）
  3. 数据完整性：通过 HMAC 防止篡改

握手过程（简化 5 步）

  1. ClientHello：客户端发送支持的 TLS 版本、加密套件、随机数 R1
  2. ServerHello + 证书：服务端选定加密套件、回传随机数 R2、发送数字证书
  3. 客户端验证证书 → 生成预主密钥 PreMaster → 用服务端公钥加密发送
  4. 双方根据 R1 + R2 + PreMaster 派生会话密钥
  5. 后续应用数据用对称密钥加密传输
📝 案例题答题模板：SSL/TLS 的应用方式（2 分）
部署在 Web 服务器或反向代理（如 Nginx），客户端与服务端通过 TLS 握手建立加密通道（即 HTTPS），后续所有传输数据加密+签名，防范传输过程中的窃听、篡改、中间人攻击。

局限：只保护"传输"阶段，不防内部威胁，不防已认证用户的越权操作。
4分析模型五个步骤
分析模型是系统分析阶段的重要产物，用 UML 描述系统的静态结构和动态行为。

五个步骤 10 分

  1. 定义概念类：从需求中识别业务实体（名词分析法）
  2. 识别类之间的关系：关联、聚合、组合、依赖、继承
  3. 为类添加职责：确定每个类的属性与方法
  4. 建立交互图：用顺序图/通信图描述类之间的动态行为
  5. 评审分析模型：检查完整性、一致性，与需求验证
口诀：找类 → 连关系 → 加职责 → 画交互 → 评审

结构图（静态）7 种

图核心用途

对象图某一时刻的实例快照

高频对比题

对比区别
顺序图 vs 通信图顺序图强调时间（纵向时间轴）；通信图强调结构（对象间空间布局）

6前后端分离 + MVP/MVVM

前后端分离架构核心思想 2 分填空

通过标准化 API（如 RESTful）将前端表现层与后端业务/数据层完全解耦，实现独立开发、独立部署、独立伸缩。
  • 前端：负责 UI 展示与交互（HTML/CSS/JS，运行在浏览器/小程序/App）
  • 后端：负责业务逻辑与数据处理，只对外暴露 API
  • 通信：JSON over HTTP/HTTPS（RESTful 或 GraphQL）

带来的好处 至少答 3 点 · 6 分

  1. 独立开发独立部署：前后端团队并行开发，发布互不阻塞
  2. 技术栈独立：前端 Vue/React/小程序；后端 Java/Go/Python
  3. 接口可复用：同一套 API 服务 Web、移动 App、小程序、第三方（智慧农业=多端）
  4. 提升用户体验：SPA、本地缓存、懒加载，响应更快
  5. 易扩展易伸缩：前端走 CDN，后端按服务横向扩展
  6. 关注点分离：职责清晰，便于维护与测试
  7. 安全性提升：后端不暴露页面渲染逻辑，减少注入面

MVP vs MVVM 对比表 7 分

维度MVPMVVM
中间层PresenterViewModel
数据同步手动同步自动更新
通信方式事件监听 / 接口回调双向绑定
模式基础观察者模式数据绑定
测试性不依赖 View，可直接单测需模拟 View 接口，测试较复杂
📝 题目候选项映射（A~J）
MVP 列：G（Presenter）/ B（手动同步）/ J（事件监听）/ F（不依赖View可单测）/ I（观察者模式）

MVVM 列：H（ViewModel）/ C（自动更新）/ D（双向绑定）/ E（需模拟View，测试复杂）/ A（数据绑定）

附加：API 网关高峰期变慢——优化措施 6 分

  1. API 网关水平扩展 + 负载均衡：增加网关实例，前置 LVS/Nginx 分流
  2. 引入缓存（Redis）：热点天气数据缓存，降低后端调用
  3. 限流与熔断（Sentinel/Hystrix）：保护后端，防级联雪崩
  4. 异步化与消息队列（Kafka/RocketMQ）：削峰填谷，把同步阻塞改异步
  5. 服务弹性伸缩：基于 CPU/QPS 自动扩缩容
  6. CDN/边缘节点：把静态资源/接口下沉到边缘
